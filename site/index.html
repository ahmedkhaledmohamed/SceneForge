Why it exists

Prompt-by-prompt generation drifts.
Projects shouldn't.

Generate six scenes as six independent prompts and each one finds its own palette, its own light, its own lens. SceneForge captures mood, palette, and lighting once at project creation and composes them into every image and video prompt. Per-scene overrides stay possible; drift doesn't.

The workflow

Six commands, start to finish.

00:00

sceneforge create "autumn morning"

Answers four questions — concept, mood, palette, lighting — and composes them into the project's style anchor. Creates a plain directory with a project.json.

00:01

sceneforge add-scenes

An LLM breaks the concept into single-image moments. You review before anything is saved, or write scenes yourself with --scene.

00:02

sceneforge generate-images

Two or three options per scene, anchor applied. Every artifact records its full prompt and model — the project file is its own generation history.

00:05

sceneforge select scene-01 2

Pick winners per scene — in the terminal, or side by side in the local web UI.

00:06

sceneforge generate-clips

Each selected still becomes a 5-second image-to-video clip. Failed jobs are recorded and picked up on the next run; nothing regenerates unless you ask.

00:14

sceneforge stitch

ffmpeg normalizes every clip (2× speed, geometry, fps), then chains crossfades with measured offsets. Out comes output/final.mp4, 720×1280, ready to post.

Local-first

A project is a folder.

No accounts, no database, no cloud dependency beyond the generation APIs you choose. State lives in project.json next to your media — the CLI and the local web UI read and write the same files, so use whichever fits the moment.

autumn-morning/
├── project.json          ← concept, style anchor, scenes, full prompt history
├── images/scene-01/      ← options; pick in the terminal or the web UI
├── clips/scene-01.mp4    ← image-to-video results, per scene
└── output/final.mp4      ← the stitched cut
